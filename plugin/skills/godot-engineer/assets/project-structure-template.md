# Godot Project Structure Template

> A recommended folder structure for a Godot 4 + C# project. Adapt to your project's needs; this is a starting point, not a law.

## Top-Level Layout

```
my-game/
├── .git/
├── .gitignore
├── .godot/                    ← Generated by Godot; gitignored
├── .vscode/                   ← VS Code settings (optional)
├── addons/                    ← Third-party plugins / Godot addons
├── assets/                    ← Source assets (textures, audio, models, fonts)
│   ├── audio/
│   │   ├── music/
│   │   ├── sfx/
│   │   └── voice/
│   ├── fonts/
│   ├── models/
│   ├── shaders/               ← Custom shader files (.gdshader)
│   ├── sprites/
│   │   ├── characters/
│   │   ├── enemies/
│   │   ├── environment/
│   │   ├── ui/
│   │   └── vfx/
│   └── textures/              ← For 3D textures separate from sprites
├── exports/                   ← Build output; gitignored
├── resources/                 ← Custom Resource files (.tres)
│   ├── data/
│   │   ├── enemies/
│   │   ├── items/
│   │   ├── levels/
│   │   └── weapons/
│   ├── materials/             ← 3D materials
│   └── themes/                ← UI themes
├── scenes/                    ← Scene files (.tscn)
│   ├── characters/
│   │   ├── enemies/
│   │   └── player/
│   ├── levels/
│   ├── managers/              ← Manager scenes (audio, save, etc.)
│   ├── ui/
│   │   ├── hud/
│   │   ├── menus/
│   │   └── widgets/
│   └── vfx/                   ← Particle and effect scenes
├── scripts/                   ← C# scripts (.cs)
│   ├── characters/
│   │   ├── enemies/
│   │   └── player/
│   ├── core/                  ← Core systems (game state, save, etc.)
│   ├── data/                  ← Data classes (POCOs, not Resources)
│   ├── managers/              ← Autoload manager scripts
│   ├── ui/
│   ├── utils/                 ← Utilities, extension methods, helpers
│   └── vfx/
├── tests/                     ← Unit tests (if using a test framework)
├── tools/                     ← Editor tools and @tool scripts
├── docs/                      ← Project documentation
├── project.godot              ← Godot project file
├── icon.svg                   ← Project icon
├── export_presets.cfg         ← Export preset config
├── default_bus_layout.tres    ← Audio bus layout
├── MyGame.csproj              ← C# project file (auto-generated)
├── MyGame.sln                 ← Solution file (auto-generated)
└── README.md                  ← Project README
```

## Key Folders Explained

### `addons/`

Third-party Godot plugins and addons. Each addon lives in its own subfolder.

```
addons/
├── godot-steam/
└── plug/
```

Most addons are GDScript-based; that's fine even in a C# project.

### `assets/`

**Source assets** — the things artists and audio designers create. Organized by type, then by category.

- **`audio/music/`** — looping background music
- **`audio/sfx/`** — short sound effects
- **`audio/voice/`** — voice lines (if any)
- **`fonts/`** — font files
- **`models/`** — 3D models (.gltf, .glb, .obj)
- **`shaders/`** — `.gdshader` files
- **`sprites/`** — 2D images, organized by category
- **`textures/`** — 3D textures (separate from 2D sprites)

For sprite sheets and atlases, the source PNGs and the Godot `.tres` atlas resources can live together or you can put atlases in `resources/`.

### `resources/`

**Custom `Resource` files** — `.tres` files that hold game data, not raw assets. Organized by type.

- **`data/enemies/`** — enemy stats as custom `EnemyData` resources
- **`data/items/`** — item definitions
- **`data/levels/`** — level metadata
- **`data/weapons/`** — weapon definitions
- **`materials/`** — 3D materials shared across meshes
- **`themes/`** — UI themes

Custom resources are how you separate data from code. Designers can edit them in the editor without touching scripts.

### `scenes/`

**Scene files** — `.tscn` files that define reusable trees of nodes. Organized by what they represent.

- **`characters/player/`** — the player and player-related scenes
- **`characters/enemies/`** — each enemy type
- **`levels/`** — level scenes
- **`managers/`** — manager scenes (game state, audio, save, scene switcher)
- **`ui/hud/`** — HUD components
- **`ui/menus/`** — menu screens
- **`ui/widgets/`** — reusable UI components (buttons, dialogs, lists)
- **`vfx/`** — particle systems and visual effects scenes

Each scene has a clear purpose and is reusable.

### `scripts/`

**C# scripts** — `.cs` files. Mirrors the scene structure when possible, so a scene's script lives in a parallel folder.

- **`characters/player/Player.cs`** corresponds to `scenes/characters/player/Player.tscn`
- **`core/`** — core systems that aren't tied to a specific scene
- **`data/`** — data classes (plain C# objects, not Godot Resources)
- **`managers/`** — autoload manager scripts
- **`utils/`** — extension methods, helpers, math utilities

You don't *have* to mirror the scene structure, but it makes the project easier to navigate.

### `tools/`

**Editor tools and `@tool` scripts** — code that runs in the Godot editor itself, not in the game. Useful for procedural authoring, asset processing, and editor plugins.

### `docs/`

**Project documentation** — design docs, ADRs, technical documents, onboarding guides. Markdown files. See documentation-writer for the writing side.

## `.gitignore`

A starting `.gitignore` for a Godot 4 + C# project:

```
# Godot 4
.godot/
.import/

# C#
bin/
obj/
*.user
*.suo
*.pdb

# Exports
exports/

# IDE
.idea/
.vs/
*.swp
.DS_Store
Thumbs.db

# Logs
*.log

# Environment
.env
.env.local
```

**Do commit:**
- `project.godot`
- `.tscn`, `.tres`, `.gd`, `.cs` files
- `.csproj`, `.sln`
- `.import` files (next to assets)
- `export_presets.cfg`
- `addons/` (third-party plugins are dependencies)

**Do NOT commit:**
- `.godot/` (cache)
- `bin/`, `obj/` (C# build output)
- `exports/` (built games)

## Naming Conventions

### Files and folders

- **`snake_case`** for asset filenames: `goblin_idle.png`, `level_1.tscn`, `sword_data.tres`
- **`PascalCase`** for C# script files: `Player.cs`, `EnemyAI.cs`, `HealthBar.cs`
- **Folders** in `snake_case`: `characters/`, `enemies/`, `power_ups/`

The convention split is awkward but practical: Godot's asset filenames are typically snake_case (matching its GDScript convention), but C# files follow .NET conventions (PascalCase).

### Scenes vs scripts

Match the scene name to its main script:

- `Player.tscn` → `Player.cs`
- `Enemy.tscn` → `Enemy.cs` (or `Goblin.tscn` → `Goblin.cs`)
- `MainMenu.tscn` → `MainMenu.cs`

This makes the relationship obvious.

### Custom resources

- `WeaponData.cs` (the class) → `sword.tres`, `axe.tres`, `bow.tres` (the data files)

The class lives in `scripts/data/`; the instances live in `resources/data/weapons/`.

## C# Project Conventions

### Project file

The `.csproj` is auto-generated by Godot:

```xml
<Project Sdk="Godot.NET.Sdk/4.2.0">
  <PropertyGroup>
    <TargetFramework>net8.0</TargetFramework>
    <EnableDynamicLoading>true</EnableDynamicLoading>
    <Nullable>enable</Nullable>
  </PropertyGroup>
</Project>
```

You can add NuGet references via `dotnet add package`.

### Namespaces

Use namespaces to mirror folder structure (optional but helpful):

```csharp
namespace MyGame.Characters.Player;

public partial class Player : CharacterBody2D
{
    // ...
}
```

This isn't required by Godot but it's idiomatic C# and helps avoid name collisions.

### Code style

Standard C# conventions with Godot-specific exceptions (`_Process`, `_Ready`, etc.). See [gdscript-vs-csharp.md](../references/gdscript-vs-csharp.md).

## Variations on the Structure

### Smaller projects

For a game jam or prototype, this structure is overkill. A flat layout works:

```
my-game/
├── assets/
├── scenes/
├── scripts/
└── project.godot
```

Don't over-organize until the project warrants it.

### Larger projects

For a project with many systems, you might add:

- **`scripts/systems/`** — top-level game systems (combat, dialogue, inventory, quests)
- **`scripts/networking/`** — multiplayer code
- **`scripts/ai/`** — AI behavior trees and state machines
- **`tests/integration/`**, **`tests/unit/`** — test organization

### Team conventions

If a team has agreed on a different structure, follow it. Consistency matters more than any specific layout. Document the structure in your project's README.
